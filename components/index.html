Components

This preliminary component library is a work in progress, likely incomplete, and has currently only been explored for mobile. Tablet and desktop considerations to follow.

If you’re interested in playing with these components, you can grab the XD file. You can also peek at the views I created for initial proof of concept.

Action Drawer

A sticky drawer to highlight a single action or progress indicator. Used on detail views. Note: The progress indicator has not yet been implemented.

<ActionDrawer
   message [string]
   messageStyle [style object]
   barStyles [style object]
   buttonText [string]
   buttonOnPress [function]
/>

Avatar

For consistency, avatars should always be circles and, if displayed with text, should be to the left of the text.

If an image isn't available — like if a person hasn't uploaded a profile picture — text (usually one or two initials) should be passed to the component instead.

<Avatar
   size [number]
   image [string]
   style [style object]
   text [string]
   textStyle [style object]
/>

Badge

To differentiate badges from buttons, badges should be tighter, lighter, and less rounded. The exception to the lighter rule are danger badges, which are designed for attention and should be used sparingly. There are no outlined badges.

<Badge
   label [string]
   style [style object]
   textStyle [style object]
>
   [Badge content]
</Badge>

Button

Buttons are one of the best opportunities for consistency. As such, if other button sizes are needed, they should be incorporated into this library rather than creating one-off instances. Additional best practices:

  • Button labels should use sentence case.
  • Plain text links should always be rendered in turquoise.
  • Actionable rows are not true buttons and follow different conventions. (See Rows.)

Card

A card always has an image and title; it can optionally include a subtitle, description, and/or badge (or button). If the card includes a description and a badge, the badge should be displayed below the description; otherwise, it should be displayed at the right.

The second example card style shown here is not currently in use. Add the secondary prop to your card to display the small image card.

<Card
   secondary
   onPress [function]
   style [style object]
   title [string]
   subtitle [string]
   image [string]
   placeholder [element]
   placeholderStyle [style object]
   badge [string]
   badgeStyle [style object]
>
   [Optional description content]
</Card>

Expanding Text

This is essentially a helper component. It adds a "read more" link to expand content inline for a string over 108 characters. (Why 108? A string that long, plus "read more", renders nicely in 2-3 lines across various mobile screen sizes.)

This is a very naive component at the moment. I could definitely see a case for expanding it to accommodate style props, among others, in the future.

<ExpandingText>[Content]</ExpandingText>

Highlight

A highlight block is a flexible content component for detail views. It always includes a title and description and can optionally include an image.

<Highlight
   title [string]
   description [string]
   image [string]
/>

Icon

This is a helper component that provides unified access to both our custom icons and a few libraries built into react-native-vector-icons.

If no icon type is passed to the component, it's assumed that we're using one of our custom icons.

<DoxyIcon
   type [string: FontAwesome, Entypo, EvilIcons, Feather, Ionicons, Material]
   name [string]
   style [style object]
/>

Image

DoxyImage is a wrapper over the FastImage component. It adds a spinner that is displayed before an image is loaded into the FastImage cache.

<DoxyImage
   resizeMode [string: cover, contain]
   source [string]
/>

Row

Rows are still a work in progress. It's to be determined whether other types of rows should be implemented within this component or as separate types.

A row should have a title and either an avatar, graphic, OR icon. It can optionally also have a subtitle and description.

Shelf

A shelf is a horizontally scrolling view with any number of shelf item components inside it. Shelf items are simple components with an optional label and one line of content (default) or a child element (multiline).

<Shelf
   style [style object]
   containerStyle [style object]
   contentContainerStyle [style object]
>
   <ShelfItem
      [multiline]
      label [string]
      labelStyle [style object]
   >
      [Item content]
   </ShelfItem>
   [...]
</Shelf>

Site Header (browser only)

By default, the site header includes the organization logo and a hamburger menu for mobile. While a hamburger menu often requires too many tradeoffs, it makes sense in this context for a few reasons:

  • Discoverabilty is not a chief concern. It’s more common to come to Church Center to accomplish a specific task (donate, register for an event, etc.) than to explore.
  • We (will) have an installed app that will provide a first-class mobile experience. The web experience is a supplement for accomplishing specific tasks (see above).
  • The number of items in the menu is highly variable. Many orgs will only have giving and profile management, for example. Or on a form, we may want to lock down the menu experience so that logging out is the only available action.

On a detail view (such as an individual group or event), we’ll also provide a “back” button to return to the category listing.

In some cases, subnavigation is needed and should be more discoverable than what a hamburger menu provides. In this case, we’ll provide a dropdown menu that doubles as the page title.

And in some other cases, more robust tools are needed, such as filtering and search. In this case, an expanding search is added, and the dropdown menu is replaced by a filter menu (which is essentially a multidimensional dropdown menu).

Typography

These five typographical components are designed to help with typographical consistency throughout the app.

If you need to create granular, one-off text styles, in most cases the best practice is to modify a PlainText component. This ensures that the other usages are reliable and consistent.

/* Heading */
<Heading
   onPress [function]
   level [number: 1, 2, 3, 4]
   style [style object]
   numberOfLines [number]
>
   [Text content]
</Heading>

/* All others */
<[PlainText | SecondaryText | TertiaryText | Footnote]
   onPress [function]
   style [style object]
   numberOfLines [number]
>
   [Text content]
</[PlainText | SecondaryText | TertiaryText | Footnote]>
